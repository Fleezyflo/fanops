## 1. Why a canary, and why upload-only

The 405 was a **routing verdict on `(method, path)`**. Report 09 replaced that pair from the OpenAPI spec
alone. **A spec is a claim about a server, not the server.** The residual risk named in report 09 §11.5 is
exactly this, and it is not closable by any further reading.

**Upload-only is the smallest experiment that closes it.** `POST /media/presign` + a signed `PUT` are the
*entire* changed surface. `POST /posts` is unchanged by this PR (report 09 §6.2 — post creation was already
correct), so calling it would add publish risk while proving nothing about the fix.

**This canary cannot publish.** Not by policy — structurally: the script never constructs a `ZernioPoster`,
never builds a payload, and never touches `/posts`. §5 pins that as an assertion, not an intention.

> **Why `testsrc2` and not a real clip:** a real clip is unreleased catalogue material. Uploading one to
> prove a transport contract would put unreleased content on a third-party host to answer a question a
> colour-bar answers identically. The server cannot tell the difference; we can.

## 3. Exact call budget — hard ceiling

| # | Call | Auth | Count | Enforced by |
|---|---|---|---|---|
| 1 | `POST https://zernio.com/api/v1/media/presign` | **Bearer** (required) | **exactly 1** | A counting wrapper around `requests.post` that **raises** on call 2 |
| 2 | `PUT <uploadUrl>` | **NONE** — the url is signed | **exactly 1** | A counting wrapper around `requests.put` that **raises** on call 2 |
| 3 | `HEAD <publicUrl>` | **NONE** | **0 or 1** — only to prove accessibility | Single call; falls back to `GET` with `stream=True` + immediate close **only** if `HEAD` returns 405 |
| — | **`POST /posts`** | — | **0 — FORBIDDEN** | The wrapper **raises** on any url containing `/posts`. **This is an assertion, not a convention** |

**Total: at most 3 HTTP requests.** **Stops after one asset** — the script has no loop and no retry.

> **No retry, deliberately.** A retry would double the call budget and could mask a flaky result as a pass.
> A transport failure is a **canary failure** (§8) to be reported, not retried.

## 4. Execution path — the REAL shipped code

The canary calls **`fanops.post.zernio.zernio_upload_media(cfg, path, account_id=None)`** — the actual
function PR #694 ships. It is not a reimplementation of the contract; a hand-rolled `requests` script would
prove the *spec* works and leave the *code* untested, which is the wrong half.

```
zernio_upload_media
  ├─ maybe_shrink_for_cap  -> no-op (asset ≪ 4 MB cap; no ffmpeg re-encode)
  ├─ POST {base}/media/presign  {filename, contentType:"video/mp4", size}   [call 1, Bearer]
  ├─ _put_signed -> PUT <uploadUrl>  raw bytes, Content-Type, NO auth       [call 2, unsigned client]
  └─ return publicUrl        (the PUT body is never parsed)
```

**Config isolation:** `Config(root=<scratch tmpdir>)` — **not** the live `FANOPS_ROOT`. The function reads
only `zernio_api_key`, `zernio_url`, `zernio_max_upload_bytes`. It opens no ledger and calls no logger.
`account_id` is unused by the presign flow (report 09 row 9), so **no account is referenced**.

## 5. Assertions enforced DURING the run (the script fails closed)

| # | Assertion | Why it is checked in-flight, not after |
|---|---|---|
| 1 | **No url containing `/posts` is ever requested** | After-the-fact detection would mean it already happened |
| 2 | **No `Authorization` header on the PUT** | Inspected on the outgoing request. A leak to third-party storage is not undoable |
| 3 | `POST` count ≤ 1, `PUT` count ≤ 1 | Hard ceiling |
| 4 | **Ledger SHA-256 identical before and after** | The strongest available proof of "no ledger mutation" — a digest, not a promise |
| 5 | **`queued == 0` and the same 4 `failed` ids, before and after** | Re-read from the live ledger read-only (`file:…?mode=ro`) |
| 6 | **No secret in any output** | Every emitted string passes through `_scrub_signed` + `redact(key)` before printing |

## 6. What is recorded

| Recorded | Redaction |
|---|---|
| Asset **SHA-256**, **MIME**, **exact byte size** | none needed — not secret |
| **presign status code**; **PUT status code**; **HEAD/GET status code** | none needed |
| **`publicUrl`** — verbatim | **Safe by design**: unsigned, intentionally public, and the value we would put in `mediaItems[]` |
| **`uploadUrl`** | **NEVER recorded.** Only its **host + path**, with the **entire query string dropped**, plus the *names* of the `X-Amz-*` params present (proving the flow is signed without emitting a signature) |
| `expiresIn`, `key` | recorded (`key` is an object path, not a credential) |
| **Any error body** | **`_evidence(cfg, resp)`** — the shipped helper: `Allow` header + `redact(_scrub_signed(text), key, limit=400)` |
| **Any transport exception** | **`_scrubbed_transport`** — the shipped helper: class + stage only for the signed PUT |
| **`ZERNIO_API_KEY`** | **Never read into a printable variable, never echoed, never written.** Loaded into `os.environ` from the live `.env` by key name only |

> The canary's redaction **is the code under test**. If a secret can leak from the canary's own output, that
> is itself a finding about `_evidence`/`_scrubbed_transport` — the canary tests its own safety claim.

## 7. Temporary-media expiry and the cleanup limitation — stated plainly

**We cannot delete the uploaded object.** No documented client-side delete for a presigned temp object was
found in the OpenAPI spec (S0) or the media guides.

| Fact | Consequence |
|---|---|
| Zernio media is **temporary (~7 days)** and becomes **permanent only when referenced by a published post** | The canary object is **never referenced** → it is never made permanent → it **expires unreferenced** |
| No delete endpoint | **The object persists until Zernio's expiry.** This is a real, accepted, bounded residual |
| The object is a **colour-bar test pattern** | The residual is: *one 2-second test-pattern file sits in the operator's own Zernio media storage for up to ~7 days.* **No private, unreleased, or catalogue content is exposed at any point** |

> This limitation is why the asset is generated rather than real. **The cleanup gap is designed around, not
> discovered afterwards.**

## 8. Criteria

### 8.1 SUCCESS — all must hold

| # | Criterion |
|---|---|
| 1 | `POST /media/presign` → **2xx** returning **both** `uploadUrl` **and** `publicUrl` |
| 2 | **Signed PUT succeeds** (2xx) |
| 3 | **`publicUrl` accessible** when checked (2xx on HEAD/GET) |
| 4 | **No `Authorization` header was sent on the PUT** — asserted on the outgoing request |
| 5 | **No secret in output or logs** — no API key, no `X-Amz-Signature`/`-Credential`/`-Security-Token`, no full `uploadUrl` |
| 6 | **`queued == 0`** after |
| 7 | **The same four `failed` ids**, unchanged, after |
| 8 | **No social post exists** — `/posts` never called |

### 8.2 FAILURE — any one aborts and is reported as-is

| # | Criterion |
|---|---|
| 1 | **Any non-2xx** on presign, PUT, or the publicUrl check |
| 2 | **Malformed presign response** — missing `uploadUrl` or `publicUrl`, or non-JSON |
| 3 | **Secret exposure** in any sink |
| 4 | **An unexpected `/posts` call** |
| 5 | **Ledger or queue mutation** — digest or counts differ |
| 6 | **An unexpected external object** beyond the single temporary upload |

**A failure is not retried and is not "fixed forward" in the same run.** It is reported with its bounded
redacted evidence, and the gate is re-presented.

> **A 405 on the PUT would be the single most valuable failure available** — it would mean the presign
> contract is *also* not what the spec says, and would invalidate report 09 §6 rather than confirm it.
